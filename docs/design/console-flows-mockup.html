运营端动线重设计

现状是"功能都有、动线没有"：任务路径靠用户脑内串联。重设计围绕三条任务动线组织： 接入（从 0 到可问答）、治理(待办驱动的日常维护)、验证（改完立刻试一问）。

现状审计 · 八处粗糙点

1 · 空状态无引导新租户打开概览是九个 0，不知道第一步做什么。
2 · SOP 是隐性的接入五步散落在按钮里，无进度感知，中断后不知道走到哪了。
3 · 原生弹窗建实体/角色用 prompt()，冷启动结果用 alert()——最粗糙的交互。
4 · 动线断点toast 说"去确认队列查看"却没有链接；校验报错"请先绑定语义角色"却不能点过去。
5 · 上下文丢失绑定选择器出现在表格外面，和被点的单元格失去空间关联。
6 · 长操作无进度冷启动/集成只有按钮文字变化，结果一闪而过没有落点。
7 · 风险不分级激活换源影响所有用户却一键即切；普通绑定反而多步。
8 · 两端割裂对话页与控制台互相到达全靠手动，语义修改后无法就地验证效果。

动线 A · 接入向导（从 0 到可问答）

五步向导替代散落按钮。状态持久化：中断后回来从断点继续；每步完成自动推进并给出下一步动作。

动线 B · 待办驱动的概览（治理收件箱）

概览从"数字堆砌"改为"待办清单"：每项待办 = 数量 + 原因 + 一键直达；处理完回到清单，像清收件箱。统计数字降级为次要信息。

动线 C · 就地验证（改完试一问）

任何语义变更的完成态都带"试一问"：不离开控制台发起一次真实问答，验证口径生效。这是语义层工作的闭环——改口径的人当场看到 agent 按新口径回答。

交互细节升级清单

1弹窗全面替换。prompt()/alert() → 内联表单与结果卡；错误信息驻留在触发位置旁，不再一闪而过。
2报错即动作。 "请先在映射矩阵为表 orders 绑定语义角色 支付日期" 附带 [去绑定] 按钮—— 点击跳转矩阵并高亮目标单元格、自动打开绑定选择器。所有跨页提示都带直达链接。
3选择器锚定单元格。矩阵绑定选择器 渲染为被点单元格正下方的展开行（accordion row），保持空间上下文；当前行高亮。
4长操作三态。进行中（行内进度： "profiling 6/10 列"）→ 完成（结果卡：产出数量 + 下一步主按钮）→ 失败（原因 + 重试）。
5风险分级确认。激活换源弹确认 （"将影响所有用户的查询目标，当前 3 个活跃会话"）；只读操作零确认。
6两端互通锚点。对话页回答的口径声明 可点击 → 直达控制台该指标编辑（带权限）；控制台"试一问"面板复用对话 API。
